# Fitness-Agent 统一开发规范

项目名称：Fitness-Agent

论文题目：《融合知识增强与动态规划的健身智能体研究——基于 Qwen2.5 大模型的实现》

本文档是 Fitness-Agent 项目的统一开发规范文件。后续所有功能开发、数据库调整、接口设计、前端页面开发和 Agent 能力扩展，均应遵守本文档。

## 1. 项目开发原则

- 所有开发必须优先服务论文创新点。
- 代码结构必须与系统模块对应。
- 保持论文章节、系统模块、代码目录三层映射。
- 系统设计应优先支撑毕业论文撰写、实验验证和答辩展示。
- 新增功能必须能够说明其在 Fitness-Agent 智能体系统中的位置和价值。

## 2. 开发前必须阅读

每次开发前，必须阅读并确认以下文档：

- `docs/thesis_mapping.md`
- `docs/module_catalog.md`
- `docs/system_architecture.md`

阅读重点：

- 当前功能对应哪个系统模块。
- 当前功能对应哪个论文章节。
- 当前功能是否影响系统架构、模块职责或数据流。

## 3. 模块开发规范

新增模块必须：

- 标注模块编号（M1~M8）。
- 标注对应论文章节。
- 标注功能说明。
- 更新 `docs/thesis_mapping.md`。
- 更新 `docs/module_catalog.md`。

模块开发时必须明确：

- 模块输入。
- 模块输出。
- 依赖模块。
- 数据流方向。
- 对应论文创新点。

## 4. 文档更新规范

每次开发后，必须更新 `docs/development_log.md`。

记录内容包括：

- 完成内容。
- 新增文件。
- 修改文件。
- 设计原因。
- 后续计划。

如果开发内容涉及模块职责、代码目录、系统架构或数据库结构变化，还必须同步更新：

- `docs/thesis_mapping.md`
- `docs/module_catalog.md`
- `docs/system_architecture.md`

## 5. 代码规范

所有文件必须包含：

- 文件头注释。
- 模块说明。
- 类说明。
- 函数说明。
- 数据流说明。

代码实现要求：

- 代码命名应清晰表达业务含义。
- 避免将跨模块逻辑写入单一文件。
- 后端业务逻辑应优先放在 service 层。
- API 层只负责参数接收、响应返回和依赖注入。
- 数据模型、Schema、Service、Router 应保持边界清晰。

## 6. 数据库规范

新增数据表必须：

- 更新数据库设计文档。
- 标注表用途。
- 标注与系统模块关系。

数据库设计要求：

- 数据库结构以项目 ORM 模型和数据库设计文档为标准。
- 不允许为了兼容临时手工表而反向修改系统领域模型。
- 新增字段必须说明业务含义和使用场景。
- 涉及用户画像、训练计划、反馈记录、知识库、模型日志的数据表必须标注对应模块编号。

## 7. FastAPI 规范

新增 API 必须：

- 标注输入参数。
- 标注返回参数。
- 标注对应模块。
- 标注对应论文章节。

接口开发要求：

- 路由文件负责 HTTP 接口定义。
- Service 文件负责业务逻辑。
- Schema 文件负责请求和响应数据结构。
- Model 文件负责数据库持久化结构。
- 接口路径应统一使用 `/api/{module}` 风格。

## 8. 前端规范

新增页面必须：

- 标注所属模块。
- 标注页面职责。
- 标注数据来源。

前端开发要求：

- 页面结构应优先服务移动端 H5。
- 页面文案应体现知识增强、动态规划、Qwen2.5、智能体决策和用户反馈闭环。
- 模拟数据应集中管理，后续便于切换为 FastAPI 接口。
- 页面、组件和路由应保持清晰目录结构。

## 9. Agent 开发规范

所有 Agent 能力开发必须优先使用：

- `AgentContext`

禁止直接跨模块读取数据库。

Agent 开发要求：

- 用户画像模块负责构建 `AgentContext`。
- 知识增强模块、动态规划模块、Qwen2.5 交互模块应通过 `AgentContext` 获取用户上下文。
- Agent 调度层负责组织模块调用，不应把模块内部实现混写在接口层。
- 新增 Agent 能力必须说明输入、输出、决策依据和可解释内容。

## 11. 执行要求

- 开发前先定位模块编号。
- 开发中保持代码、文档、论文映射同步。
- 开发后必须更新日志并进行基本验证。
- 任何破坏三层映射关系的改动，都应先调整文档再调整代码。
