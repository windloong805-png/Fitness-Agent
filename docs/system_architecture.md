# Fitness-Agent 系统架构说明

论文题目：《融合知识增强与动态规划的健身智能体研究——基于 Qwen2.5 大模型的实现》

本文档描述 Fitness-Agent 的系统分层、模块关系和核心数据流，用于指导后续工程实现、论文架构图绘制和答辩展示。

## 总体架构

```text
┌──────────────┐
│     用户     │
└──────┬───────┘
       │ 输入目标、状态、反馈、问题
       ▼
┌──────────────┐
│  移动端 App  │
│ Vue3 H5 原型 │
└──────┬───────┘
       │ REST API / WebSocket
       ▼
┌──────────────┐
│ FastAPI 后端 │
└──────┬───────┘
       │ 统一封装请求、鉴权、数据读写
       ▼
┌──────────────┐
│ Agent 调度层 │
└──────┬───────┘
       │ 构建 AgentContext，调度各能力模块
       ▼
┌──────────────┐
│ 用户画像模块 │
└──────┬───────┘
       │ 用户长期画像 + 动态状态画像
       ▼
┌──────────────┐
│ 知识增强模块 │
│     RAG      │
└──────┬───────┘
       │ 检索运动处方、动作安全、恢复知识
       ▼
┌──────────────┐
│ 动态规划模块 │
└──────┬───────┘
       │ 计算训练收益、疲劳代价、最优路径
       ▼
┌──────────────┐
│ Qwen2.5 模块 │
└──────┬───────┘
       │ 生成解释、问答回复、计划调整说明
       ▼
┌──────────────┐
│ 反馈闭环模块 │
└──────┬───────┘
       │ 回收完成度、疲劳感、采纳率、RPE
       ▼
┌──────────────┐
│ MySQL 数据库 │
└──────────────┘
```

## 分层说明

| 层级 | 组成 | 说明 |
|------|------|------|
| 表现层 | `frontend/` | 移动端优先的 Vue 3 H5 页面，展示驾驶舱、训练计划、AI 教练、数据分析和用户设置。 |
| 服务层 | `backend/` | 计划用于 FastAPI 接口，承接前端请求、组织业务流程、连接数据库与模型服务。 |
| Agent 调度层 | `backend/agent/`（规划目录） | 将用户请求转化为 AgentContext，调度画像、RAG、动态规划、Qwen2.5 和反馈闭环模块。 |
| 知识与模型层 | `rag/`, `knowlege_base/`, `qwen/` | 提供知识增强检索、提示词构造、大模型交互和可解释回答。 |
| 决策与实验层 | `dynamic_planning/`, `experiments/` | 实现训练收益评估、动态规划路径搜索、算法对比和实验评价。 |
| 数据层 | `datasets/`, MySQL | 存储用户画像、训练计划、反馈数据、实验数据和知识库元数据。 |

## 核心数据流

### 1. 训练推荐数据流

```text
用户状态输入
  -> 移动端 App
  -> FastAPI 后端
  -> Agent 调度层
  -> 用户画像模块生成 AgentContext
  -> 知识增强模块检索约束规则
  -> 训练收益评估模块计算候选动作收益
  -> 动态规划模块生成训练路径
  -> Qwen2.5 模块生成推荐理由
  -> 前端展示今日训练与 7 天计划
```

### 2. AI 问答数据流

```text
用户自然语言问题
  -> AI 教练页
  -> FastAPI 后端
  -> Agent 调度层
  -> RAG 检索相关健身知识
  -> 读取用户画像与近期反馈
  -> Qwen2.5 生成回答
  -> 前端展示聊天结果
```

### 3. 用户反馈闭环数据流

```text
训练完成度 / 主观疲劳 / RPE / 采纳情况
  -> 前端反馈入口
  -> FastAPI 后端
  -> 反馈闭环模块
  -> MySQL 数据库存储
  -> 更新用户动态状态画像
  -> 影响下一次动态规划推荐
```

### 4. 实验评价数据流

```text
历史训练数据 / 模拟用户数据
  -> experiments/
  -> 对比不同推荐策略
  -> 评估收益、疲劳控制、采纳率和解释质量
  -> 支撑论文实验章节
```

## 架构设计原则

- 前端与后端解耦：当前前端使用模拟数据，后续通过 API 接入 FastAPI。
- 模块边界清晰：画像、RAG、动态规划、Qwen2.5 和反馈闭环均可独立迭代。
- 论文映射可追踪：每个模块都应能对应到论文章节、代码目录和创新点。
- 可解释性优先：推荐结果不仅给出训练计划，还要说明知识依据、决策过程和用户反馈影响。
- 长期可扩展：预留 uni-app、多模态动作识别、营养推荐和解释性评估扩展空间。
